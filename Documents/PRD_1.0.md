## 0. 产品哲学

> **AI 是一个积极参与的协作者，但用户永远能看见 AI 在想什么，并且能纠正它。**

这句话是 Forgify 所有设计决策的判断标准。当出现模棱两可的设计选择时，以此为准。

---

## 1. 产品定位

### 1.1 一句话描述

Forgify 是一个本地运行的桌面应用，让普通人通过对话，把每天重复做的事情锻造成可以自动运行的流程。

### 1.2 名字的含义

**Forge**（锻造）+ **-ify**（...化）——让一切可被锻造。

用户在对话中一点点锻造属于自己的工具和流程。这不是拼装现成零件，而是从原料（需求）开始，在炉火（对话）中反复塑形，最终得到一件趁手的武器。

### 1.3 解决什么问题

现有 AI 工具的三种形态都有局限：

- **对话型 AI（ChatGPT 等）**：给答案，但不能"执行"；用户无法把多次对话的能力沉淀成可复用的流程。
- **可视化编排平台（Dify、n8n）**：功能强大，但门槛高；工具箱固定，无法在对话中即时创造新工具。
- **代码型 Agent（Claude Code）**：面向开发者，终端交互；非技术用户无法使用，且缺乏流程编排能力。

Forgify 填补中间地带：**让非开发者也能通过自然对话，创造属于自己的工具和自动化流程，并对整个过程保持可见和可控。**

### 1.4 目标用户

**主要用户（P0）**：

- **有重复性工作的白领**：运营、行政、财务、采购、市场等岗位，每天处理邮件、Excel、报表、跨系统数据搬运。
- 特征：知道 AI 有用，但不知道如何把多个 AI 能力组合起来；有耐心探索；愿意为自己造工具。

**次要用户（P1）**：

- **想快速自动化的开发者**：懂一点代码，但不想为每个小需求都写完整项目。希望快速把想法落成可运行的流程。
- 特征：能看懂代码，会给 AI 提更精确的要求，是高级工具的创造者和分享者。

### 1.5 核心价值 (First Win)

> **用户看到一个复杂的目标，被 AI 拆解成一个一个清晰的节点，然后通过对话把每个节点攻克，最后串联起来，变成一个真正能跑的流程。**

这个"拆解 → 逐个攻克 → 串联"的过程，就是 Forgify 的魔法时刻。

## 2. 核心概念模型

Forgify 包含四个核心对象：**对话、工具、工作流、主题**。另有一个辅助概念：**文件夹**。

### 2.1 对话 (Chat)

所有创造的起点和入口。

**特征**：

- 每个对话是一个独立的会话，有完整上下文历史
- 对话可以创建、修改、删除工具和工作流
- 对话可以触发已有工作流运行，或调整工作流配置
- 一个对话可以关联多个工作流和工具
- 对话本身是独立的资产，可以被命名、归档、删除

**对话的两种形态**：

- **自由对话 (Free Chat)**：用户开启的通用对话，可以聊任何事，通常是所有创造的起点
- **专属对话 (Linked Chat)**：附着在某个工具或工作流上的对话，上下文被限定在这个资产的范围内

### 2.2 工具 (Tool)

一个规范化的 Python 文件，封装一个具体的功能。

**工具的本质**：每一个工具都是一个可以被对话锻造出来的函数。

**文件结构**：

```
my_tool/
├── tool.yaml         # 元数据和输入输出声明
├── main.py           # AI 生成的实现代码
├── requirements.txt  # Python 依赖
└── tests/            # AI 生成的测试用例
    ├── test_cases.yaml
    └── sample_data/
```

**tool.yaml 规范**：

```yaml
schema_version: "1.0"
name: "处理供应商报价Excel"
id: "tool_abc123"
description: "从Excel文件中提取报价数据，标准化为统一格式"
author: "用户A"
created_at: "2026-04-19T10:00:00Z"
version: "1.2"

permissions:
  - file_read
  - file_write

inputs:
  - name: file_path
    type: string
    required: true
    description: "Excel文件的本地路径"
  - name: vendor_name
    type: string
    required: false
    default: ""
    description: "供应商名称，用于特殊格式识别"

outputs:
  - name: result
    type: object
    schema:
      items: array
      total_amount: number
      currency: string
    description: "标准化后的报价数据"

linked_chat_id: "chat_xyz789"  # 关联的专属对话
```

**工具详情页的交互**：

- 左侧：输入输出声明（可视化表单，非代码）
- 中间：测试运行区，可输入样例数据，实时查看结果
- 右侧：专属对话，用户与 AI 讨论这个工具的实现和调整
- 顶部切换标签：`接口定义` / `测试运行` / `查看代码`
- 代码对用户开放，但不是主要交互面板

**工具的生命周期**：

1. `draft`：对话中刚被创建，未测试
2. `testing`：用户正在测试，未确认
3. `ready`：用户确认可用，可被工作流引用
4. `deprecated`：被用户标记为废弃

**工具的分享**：

- v1 版本：导出为 `.forgetool` 文件（本质是压缩包），拖拽到另一台安装了 Forgify 的电脑即可导入
- 导入时，Forgify 读取 tool.yaml 并展示工具信息和权限需求，用户确认后加入工具库
- 未来版本考虑社区市场，v1 不做

**内置工具（第一版）**：

覆盖办公自动化的高频场景：

| 类别 | 工具 |
|---|---|
| 邮件 | Gmail 读/发、Outlook 读/发、邮件附件下载、邮件过滤器 |
| 文件 | Excel 读/写、Word 读/写、PDF 读/生成、CSV 处理、本地文件读写 |
| 日历 | Google Calendar、Outlook Calendar、日程创建/查询 |
| 通讯 | 飞书、钉钉、Slack、企业微信、Telegram |
| 网络 | HTTP 请求、网页抓取、RSS 订阅、Webhook 接收 |
| 数据 | JSON 处理、正则提取、数据转换、去重/排序/过滤 |
| 系统 | 定时触发、文件夹监控、剪贴板读写、桌面通知 |
| AI | LLM 调用（支持自定义 API Key）、文本总结、翻译、分类 |

### 2.3 工作流 (Workflow)

节点和连线组成的图，每个节点调用一个工具, 或者是与AI的交互

**工作流的本质**：把多个工具按一定顺序和条件串联起来，变成一个可反复运行的流程。

**节点类型**：

| 类型 | 说明 |
|---|---|
| **触发节点** | 工作流的起点：手动触发 / 定时触发 / 事件触发 / Webhook |
| **工具节点** | 调用一个工具 |
| **条件节点** | if / else / switch，根据上游数据决定走向 |
| **循环节点** | 遍历列表，对每一项执行内部子流程 |
| **并行节点** | 同时执行多个分支，等待全部完成后汇合 |
| **LLM 节点** | 直接调用大模型，用于判断、总结、生成 |
| **人工确认节点** | 暂停流程，等待用户在通知中确认 |
| **变量节点** | 定义中间变量或常量 |
| **子工作流节点** | 调用另一个工作流 |

**工作流的生命周期**：

1. `draft`：在对话中正在构建
2. `ready`：所有节点配置完整，可运行
3. `deployed`：已部署到生产环境，可被定时或事件触发
4. `paused`：已部署但暂停触发
5. `archived`：归档，不再使用

**工作流详情页**：

- 画布为主，节点和连线可拖拽
- 顶部工具栏：`运行一次` / `部署` / `暂停` / `查看运行历史`
- 节点可点击展开，查看输入输出、当前配置
- 节点可标记为"缺失"状态（工具尚未创建），高亮显示

**部署配置**：

- **触发方式**：手动 / 定时（cron 表达式，有可视化编辑器）/ 事件 / Webhook
- **错误处理**：失败重试次数、重试间隔、失败后动作（通知 / 停止 / 跳过）
- **通知设置**：成功通知、失败通知、通知渠道（站内 / 邮件 / IM）
- **权限确认**：部署前必须再次确认所有工具的权限

### 2.4 主题 (Topic)

AI 和用户共同维护的语义空间。

**主题的双重身份**：

- **作为线**：穿过所有相关资产的语义标签，随时可感知
- **作为房间**：可以走进去的聚焦空间，只显示这个主题相关的内容

**主题的组成**：

```yaml
topic:
  id: "topic_email_automation"
  name: "邮件自动化"
  goal: "自动处理供应商报价邮件，提取数据，格式化后转发给采购团队"  # 用户可编辑的一句话目标
  created_at: "2026-04-10"
  last_active: "2026-04-19"

  memory:
    context: |
      用户使用 Gmail，公司邮箱后缀是 @company.com
      主要处理三家供应商的报价：供应商A、B、C
      Excel 格式每家不同，需要不同的解析逻辑
      处理后需要统一格式，按金额排序，发给 purchase@company.com
    decisions:
      - "2026-04-12：决定使用金额排序而不是供应商排序，因为采购更关注总额"
      - "2026-04-15：取消自动发送，改为人工确认后发送"

  linked_assets:
    chats: [chat_001, chat_003, chat_007]
    tools: [tool_excel_parser_a, tool_excel_parser_b, tool_format_unify]
    workflows: [workflow_daily_quote_process]

  gaps:
    - "还没处理供应商D的报价格式"
    - "失败时的错误通知还没配置"
```

**主题的创建方式**：

- **AI 主动建议**：当一次对话产生了工具或工作流，AI 判断这件事足够具体，主动提出"要不要把这个加到'邮件自动化'主题下？"
- **用户主动创建**：在 Home 页可以手动创建主题
- **从资产反向创建**：在任意资产上，用户可以"归入主题 / 新建主题"

**主题的入口**：

- 主题不是独立的导航 tab
- 主题入口在 Home 页，以卡片形式陈列
- 任何资产上都有标签显示所属主题，点击标签可跳转进入主题房间

**进入主题房间**：

- 布局与主界面一致（三栏布局），但内容被过滤
- 左边栏只显示这个主题下的资产
- 中间对话默认是"主题对话"，AI 已加载主题记忆，不需重新解释背景
- 右边画布默认显示这个主题的资产关系图
- 顶部显示主题的一句话目标，用户可编辑
- 退出主题房间回到全局视图

**主题记忆的透明性**：

- 用户可随时查看主题的完整记忆（context + decisions）
- 用户可编辑、删除任何一条记忆
- AI 每次更新主题记忆时，用户会在主题房间内看到一条"AI 更新了主题记忆"的提示，可点开查看

### 2.5 文件夹 (Folder)

用户自己的组织方式，与主题完全独立。对话，工具，工作流都可以被放到文件夹里，但是这只是用户归档的一种方式。

**与主题的区别**：

| | 文件夹 | 主题 |
|---|---|---|
| 维护者 | 用户 | AI + 用户共同维护 |
| 性质 | 物理组织 | 语义索引 |
| 归属 | 一个资产只能在一个文件夹 | 一个资产可以属于多个主题 |
| 目的 | 收纳 | 聚焦 |

文件夹是可嵌套的，支持多层结构。对话、工具、工作流各自有独立的文件夹树。

## 3. 界面结构

### 3.1 整体导航

底部 tab 栏（或侧边栏，待视觉设计）包含四个主要入口：

- **Home**：抬头视角，AI 整理的全局
- **对话**：所有对话的列表与入口
- **工具**：工具库
- **工作流**：工作流库（含生产环境监控）

主题不是独立 tab，而是从 Home 进入的房间。

### 3.2 Home 页

Home 页是有观点的首页，不是空白欢迎页。内容包括：

**顶部：问候与状态**

- "早上好" / "下午好"（根据时间）
- 一句当天摘要："今天已有 3 个工作流成功运行，1 个失败需要处理"

**第一区块：需要你关注的**

- 失败的工作流运行（可点击直接进入错误修复对话）
- AI 发现的缺口（"你的邮件自动化主题还有一个未解决的问题"）
- 等待人工确认的节点

**第二区块：主题卡片**

- 以卡片形式展示所有主题
- 每张卡片：主题名、一句话目标、最近活动、包含资产数量
- 点击进入主题房间
- 有"+ 新建主题"按钮

**第三区块：最近活动**

- 最近打开的对话 / 编辑的工具 / 运行的工作流
- 按时间倒序

### 3.3 对话页

三栏布局，是所有创造的核心界面。

**左栏（约 240px）**：

- 对话列表（按文件夹组织）
- 搜索框
- 可切换显示"所有对话 / 主题下对话 / 附属对话"

**中栏（自适应）**：

- 对话消息流
- 底部输入框，支持：
  - 文本输入
  - 文件拖拽上传
  - @ 符号引用已有的工具、工作流、主题
  - 斜杠命令（`/new-tool`、`/new-workflow`、`/run`）
- 消息类型：
  - 用户文本消息
  - AI 文本回复
  - AI 操作卡片（"我创建了工具 X"）
  - 画布操作回声（"你在画布上删除了节点 Y"）
  - 工具测试结果卡片
  - 工作流运行卡片

**右栏（可展开/收起，约 50% 宽度）**：

- 画布：显示当前对话正在创造或编辑的工作流
- 如果当前对话没有关联工作流，画布为空，显示 "与 AI 聊一聊，开始锻造你的第一个流程"
- 画布顶部有标题（工作流名称），可切换"画布视图 / 列表视图"

**画布的交互**：

- 节点可拖拽移动、点击展开、右键菜单
- 连线可拖拽创建、点击删除
- 节点有状态颜色：蓝色（配置完整）、黄色（缺失工具）、红色（错误）、灰色（禁用）
- 用户在画布上的每一个操作，会生成一条"画布操作回声"注入对话上下文
- 每个操作可附加备注（小输入框，可选填），帮助 AI 理解操作意图

**工具聊天视图（特殊模式）**：

- 当用户点击画布上一个"缺失工具"的节点，界面切换为"工具锻造模式"
- 左边变成画布（缩小），右边变成工具的专属对话
- 用户与 AI 讨论这个工具的输入输出和实现
- 完成后，回到正常的三栏布局，该节点变为蓝色

### 3.4 工具页

**列表视图**：

- 工具库，按文件夹或标签筛选
- 每个工具卡片：图标、名称、描述、状态、最后使用时间
- 支持搜索、排序

**详情视图**（点击工具）：

- 上面描述过的三区布局：接口定义 / 测试运行 / 查看代码 + 专属对话
- 顶部有"在对话中使用"按钮，跳转到新的对话并引用此工具

### 3.5 工作流页

**列表视图**：

- 工作流库，按状态筛选（全部 / 草稿 / 已部署 / 已暂停）
- 每个卡片：名称、状态、触发方式、最近运行、成功率
- 已部署的工作流有"生产徽章"

**详情视图**：

- 画布为主
- 右侧可展开"运行历史"面板
- 顶部操作：运行一次 / 部署配置 / 暂停 / 克隆 / 归档

**运行历史**：

- 时间线视图，每一次运行一条记录
- 每条记录可展开，看每个节点的输入、输出、耗时、状态
- 失败的运行高亮显示，点击进入错误修复对话

### 3.6 主题房间（从 Home 进入）

三栏布局，与主对话页相似，但内容被主题过滤。

**顶部横条**：

- 主题名 + 一句话目标（可编辑）
- "退出主题"按钮
- "查看主题记忆"按钮

**左栏**：

- 只显示这个主题下的对话、工具、工作流
- 有"添加资产到此主题"按钮

**中栏**：

- 默认打开"主题对话"（AI 已加载主题记忆的对话）
- 也可以打开这个主题下的任何具体对话

**右栏**：

- 默认显示主题关系图（所有资产之间的连接）
- 也可以像主对话页一样显示具体的工作流画布

## 4. 核心交互流程

### 4.1 首次使用引导

用户第一次打开 Forgify：

1. 欢迎页，一句话说明 Forgify 是什么
2. 让用户连接至少一个模型 API（OpenAI / Anthropic / 自定义）
3. 引导用户在 Home 页点击"创建你的第一个自动化"
4. 进入一个预设的对话，AI 问："你最想自动化哪件每天重复的事？"
5. 用户描述，AI 拆解，开始锻造

### 4.2 核心流程：从对话到生产

**场景**：用户想每天自动处理供应商报价邮件。

**步骤 1：表达意图**

用户在自由对话中说："我想每天自动处理供应商发来的报价 Excel，汇总后发给采购团队。"

**步骤 2：AI 拆解需求**

AI 回复，提出关键问题：
- 供应商邮件有特征吗（发件人、主题、关键词）？
- Excel 格式每家是否一致？
- 汇总的格式是什么？
- 发给谁？自动发还是人工确认？

用户逐一回答。

**步骤 3：工作流生成**

AI 在画布上开始生成工作流节点：

1. `定时触发`（每天 9:00）
2. `读取 Gmail`（过滤条件：特定发件人、主题关键词）
3. `下载附件`（Excel 文件）
4. `[缺失] 解析供应商 A 的 Excel`
5. `[缺失] 解析供应商 B 的 Excel`
6. `[缺失] 解析供应商 C 的 Excel`
7. `汇总数据`（内置工具）
8. `格式化 Excel`（内置工具）
9. `人工确认节点`
10. `发送邮件`

画布实时显示生成过程，用户可以看到节点一个个出现。

**步骤 4：锻造缺失工具**

用户点击节点 4（解析供应商 A 的 Excel）。界面切换到工具锻造模式。

AI 说："我需要了解供应商 A 的 Excel 长什么样。能上传一个样例给我吗？"

用户拖入样例文件。AI 分析结构，与用户确认：
- 表头在第 2 行
- 价格列是 C 列
- 数量列是 D 列
- 需要过滤税率列为空的行

用户确认。AI 生成代码，自动运行测试。测试通过，工具状态变为 `ready`，节点变为蓝色。

用户重复此流程，完成节点 5、6。

**步骤 5：整体测试**

用户点击工作流顶部的"运行一次"。

工作流按节点顺序执行，每个节点的输入输出实时显示在画布上。

到人工确认节点，Forgify 弹出桌面通知，用户点击"查看内容 → 确认发送"。

整体运行成功。

**步骤 6：部署**

用户点击"部署"。

部署弹窗依次确认：
- 触发方式：每天 9:00 ✓
- 权限再次确认：读取 Gmail、下载附件、发送邮件 ✓
- 错误处理：失败重试 2 次，间隔 5 分钟，失败后站内通知 ✓
- 通知设置：成功静默、失败通知 ✓

用户确认。工作流进入 `deployed` 状态。

**步骤 7：主题整理**

AI 在对话中提示："我注意到这个工作流和'邮件处理'相关。要不要建一个'供应商报价自动化'主题？"

用户同意。Forgify 创建主题，归入相关对话、工具、工作流，并生成初始的主题记忆。

### 4.3 错误处理流程

**场景**：部署后的工作流运行失败。

**步骤 1：通知**

失败发生，用户同时在三个地方收到通知：
- 桌面通知弹窗
- Home 页"需要你关注的"区块
- （如配置）邮件通知

**步骤 2：错误诊断**

用户点击通知，直接进入**错误修复对话**。

这不是打开日志，而是一个新对话，AI 已经分析完毕，开场消息是：

> "昨晚 9:00 的供应商报价处理失败了。原因是供应商 A 的 Excel 格式变了，多了一列'税率'。
>
> 我有三种处理方式：
> 1. 自动修改解析工具以适应新格式（推荐）
> 2. 你和我一起讨论怎么改
> 3. 先跳过这次，保持工具不变"

**步骤 3：选择修复方式**

用户选择方式 1。AI 修改工具代码，运行测试，测试通过，让用户确认。

用户确认。工具版本从 1.0 升级到 1.1。工作流恢复正常。

**步骤 4：记忆更新**

AI 在主题记忆里追加一条："2026-04-19：供应商 A 的 Excel 增加了税率列，工具已适配。"

### 4.4 工具分享流程

**场景**：用户 A 把自己锻造的工具分享给用户 B。

**用户 A 侧**：

- 在工具详情页点击"导出"
- 选择是否包含测试用例和样例数据
- 生成 `.forgetool` 文件（例如 `excel_parser_vendor_a.forgetool`）
- 通过任何方式发送给用户 B（IM、邮件、U 盘）

**用户 B 侧**：

- 把 `.forgetool` 文件拖入 Forgify 窗口
- Forgify 弹出导入确认窗口：
  - 工具名称、描述、作者
  - 输入输出声明
  - 所需权限（高亮显示）
  - 代码预览（可点开）
- 用户 B 确认，工具加入他的工具库
- Forgify 自动在工具上标注 `来源：用户 A 分享`

## 5. 权限与安全

### 5.1 权限分级

所有工具在创建时必须声明所需权限，分三级：

**读取级 (Read)**：

- 读取邮件、日历、文件、剪贴板
- 查询 API、抓取网页

**修改级 (Modify)**：

- 写入本地文件、修改日历事件
- 创建邮件草稿（不发送）
- 修改已有数据

**执行级 (Execute)**：

- 发送邮件 / 消息
- 调用付费 API
- 删除数据
- 执行任意 shell 命令
- 调用外部系统产生不可逆操作

### 5.2 权限确认时机

- **工具创建时**：AI 必须在 tool.yaml 中声明权限，用户首次测试前确认
- **工作流部署时**：部署弹窗中再次列出所有工具的权限总和，用户确认
- **执行级操作运行时**：默认"每次都确认"，用户可设置"信任此工具"跳过

### 5.3 敏感数据处理

- **API Key 管理**：所有 API Key 加密存储在本地，不离开用户电脑
- **工具分享时的敏感数据**：导出 `.forgetool` 时自动剥离 API Key、令牌、私密数据
- **日志脱敏**：工具运行日志自动识别并遮盖疑似密钥、密码、身份证号

### 5.4 运行沙箱

- 每个工具在独立的 Python 虚拟环境中运行
- 文件系统访问默认限制在用户指定的工作目录
- 网络访问遵循声明的权限，发现未声明的外部请求时暂停并通知用户

## 6. 模型与 API

### 6.1 模型支持

Forgify 支持几乎所有主流 LLM API：

- OpenAI / Azure OpenAI
- Anthropic Claude
- Google Gemini
- DeepSeek
- Moonshot / Kimi
- 通义千问、文心一言、智谱
- 任何 OpenAI 兼容的 API 端点
- Ollama 本地模型

### 6.2 Key 配置

- 用户在设置页配置各家的 API Key
- 可为不同场景指定不同模型：
  - 主对话模型（推理能力强）
  - 工具代码生成模型（代码能力强）
  - 工作流中 LLM 节点的默认模型
- 所有 Key 加密存储，永不外传

### 6.3 模型调用策略

- 首次使用引导用户配置至少一个模型
- 如果当前选用模型失败（网络、配额、限流），自动降级到备用模型
- 显著操作（工具生成、部署前代码审查）默认使用最强推理模型

## 7. 本地架构要点

### 7.1 运行方式

- Forgify 是一个桌面应用（Electron 或 Tauri，待定）
- 启动后常驻后台，状态栏有图标
- 关闭窗口 ≠ 退出应用，后台继续运行定时任务
- 真正退出需从状态栏菜单选择"退出 Forgify"

### 7.2 数据存储

- 所有数据存储在用户本地（SQLite + 文件系统）
- 默认存储路径：
  - macOS: `~/Library/Application Support/Forgify/`
  - Windows: `%APPDATA%\Forgify\`
  - Linux: `~/.config/Forgify/`
- 用户可在设置中修改存储路径

### 7.3 后台调度

- 内置调度器负责触发定时任务和事件监听
- 电脑睡眠时任务暂停，唤醒后补跑错过的任务（可配置）
- 多个工作流并发执行，有资源限制（最大并发数可配置）

### 7.4 更新机制

- 自动检查更新，用户确认后下载安装
- 内置工具与 Forgify 主程序分离，内置工具可独立增量更新

- 用户平均拥有主题数量
- 用户锻造的自定义工具数量占其使用总工具数的比例
- 工作流的平均节点数与平均运行时长

## 11. 未决事项

以下问题在 PRD 下一版本前需要决定：

1. **产品外观语言**：温暖手作风还是工业精密风？"锻造"这个隐喻在 UI 上如何体现？
2. **Tauri vs Electron**：性能 vs 开发效率的权衡
3. **工具代码的隔离程度**：多大程度信任 AI 生成的代码？是否需要静态分析层？
4. **错误修复对话的独立程度**：是新开一个对话，还是在原对话基础上延续？
5. **主题自动归类的激进程度**：AI 多久建议一次？建议频率如何避免打扰？
6. **离线能力**：没有网络时，Forgify 能做什么？（应至少能运行已部署的本地工作流）
